## Code Standards

### TypeScript (API Layer)

The DBaaS API is written in TypeScript and follows these standards:

**General**:
- TypeScript strict mode enabled (`"strict": true` in tsconfig.json).
- All functions must have explicit return types.
- All exported interfaces and types must have JSDoc documentation.
- No `any` types except in gRPC client bindings and database row mapping (use `unknown` where possible).
- Use `const` by default; use `let` only when mutation is required.

**Naming Conventions**:
- Files: `kebab-case.ts` (e.g., `backup-manager.ts`, `rate-limiter.ts`).
- Classes: `PascalCase` (e.g., `BackupManager`, `ErpAiddStrictProfile`).
- Interfaces: `PascalCase` (e.g., `ServiceInstance`, `ProvisionRequest`).
- Enums: `PascalCase` with `PascalCase` members (e.g., `DBEngine.YugabyteDB`).
- Functions and methods: `camelCase` (e.g., `validateEngine`, `triggerBackup`).
- Constants: `UPPER_SNAKE_CASE` (e.g., `RATE_LIMITS`, `CACHE_TTL`).
- Database column mapping: Use `snake_case` for database fields, `camelCase` for TypeScript fields.

**Import Order**:
1. Node.js built-in modules.
2. Third-party modules (express, pg, zod, etc.).
3. Internal modules (types, controllers, policies, etc.).
4. Separate groups with a blank line.

**Error Handling**:
- All route handlers must be wrapped in `try/catch` and call `next(err)` in the catch block.
- Use Zod's `safeParse` for request validation; never throw on validation errors.
- Return structured error responses with `success: false`, `error` code, and `message`.
- Include `requestId` in all error responses.

**Example route handler**:

```typescript
instancesRouter.get('/:id', async (req: Request, res: Response, next: NextFunction) => {
  try {
    const tenantId = req.tenantContext!.tenantId;
    const { id } = req.params;

    const result = await pool.query(
      'SELECT * FROM service_instances WHERE id = $1 AND tenant_id = $2',
      [id, tenantId],
    );

    if (result.rows.length === 0) {
      return res.status(404).json({
        success: false,
        error: 'not_found',
        message: `Instance ${id} not found.`,
      });
    }

    return res.json({
      success: true,
      data: result.rows[0],
      requestId: req.requestId,
    });
  } catch (err) {
    next(err);
  }
});
```

### Go (Gateway)

The Go gateway follows these standards:

- Go 1.22+ with module support.
- `go vet` and `golint` must pass with zero warnings.
- All exported functions must have doc comments.
- Use `envOrDefault()` for environment variable access with fallback values.
- HTTP handlers must validate HTTP method and return 405 for unsupported methods.
- Use structured logging with `log.Printf` using key=value format.
- Server configuration must include all timeout fields (`ReadHeaderTimeout`, `ReadTimeout`, `WriteTimeout`, `IdleTimeout`).

### SQL (Migrations)

- Migrations are numbered sequentially: `001_initial_schema.sql`, `002_add_indexes.sql`, etc.
- All migrations must be idempotent (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`).
- Use `UUID` primary keys generated by `gen_random_uuid()`.
- Use `TEXT` for tenant_id (not UUID) for consistency with other ERP modules.
- Use `TIMESTAMPTZ` for all timestamp columns (not `TIMESTAMP`).
- Include `created_at` and `updated_at` columns on all mutable tables.
- Define indexes for all foreign key columns and commonly filtered columns.

### YAML (CRDs and Kubernetes Manifests)

- All CRDs use the `dbaas.businessactivation.cloud` API group.
- CRD names follow the pattern: `{plural}.dbaas.businessactivation.cloud`.
- Include `additionalPrinterColumns` for all CRDs to improve `kubectl` output.
- Use `shortNames` for all CRDs (e.g., `si` for ServiceInstance, `bp` for BackupPolicy).
- Include `description` fields on all CRD spec properties.

## Branching and Commit Strategy

### Branch Naming

- Feature branches: `feature/dbaas-{ticket}-{short-description}` (e.g., `feature/dbaas-142-scylladb-adapter`)
- Bug fix branches: `fix/dbaas-{ticket}-{short-description}` (e.g., `fix/dbaas-187-quota-race-condition`)
- Documentation branches: `docs/dbaas-{short-description}`

### Commit Messages

Follow Conventional Commits format:

```
<type>(<scope>): <subject>

<body>

<footer>
```

Types: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `ci`, `perf`.

Scopes: `api`, `gateway`, `operator`, `plugin`, `policy`, `backup`, `credential`, `metering`, `quota`, `crd`, `infra`.

Examples:
```
feat(api): add ScyllaDB operator adapter

Implements the Scylla Operator adapter for provisioning, scaling,
and decommissioning ScyllaDB clusters via the Scylla Operator CRDs.

Closes: DBAAS-142
```

```
fix(quota): prevent negative instance count on decommission

Uses GREATEST(current_instances - 1, 0) instead of
current_instances - 1 to prevent the counter going below zero
when race conditions occur during concurrent decommissions.

Fixes: DBAAS-187
```

## Adding a New Engine Operator

To add support for a new database engine, follow these steps:

### Step 1: Verify AIDD Approval

Before any code changes, confirm the engine is AIDD-approved:

- If the engine is for ERP internal use, it must be on the AIDD-approved list and added to the `ErpAiddStrictProfile.ALLOWED_ENGINES` set.
- If the engine is for commercial customers only, it should be added to the `CommercialFlexibleProfile.ALLOWED_ENGINES` set and the `CommercialEngine` enum.
- If the engine is for plugin-based extension only, no core changes are needed.

### Step 2: Add Type Definitions

Edit `/services/dbaas-api/src/types/index.ts`:

```typescript
// Add to the appropriate enum
export enum DBEngine {
  // ... existing engines
  NewEngine = 'newengine',
}

// Or for commercial engines:
export enum CommercialEngine {
  // ... existing engines
  NewEngine = 'newengine',
}
```

### Step 3: Create the Operator Adapter

Create a new file at `/services/dbaas-api/src/operators/{engine}-adapter.ts`:

```typescript
import { BaseOperator, ProvisionConfig, ScaleConfig } from './base-operator';

/**
 * {EngineName} Operator Adapter
 *
 * Manages {EngineName} instances via the {OperatorName} Kubernetes operator.
 * Operator: {operator-name}
 * CRD API Group: {api-group}
 */
export class NewEngineAdapter extends BaseOperator {
  readonly engineName = 'newengine';
  readonly operatorName = '{operator-name}';

  async provision(config: ProvisionConfig): Promise<void> {
    // 1. Generate engine-specific CRD spec
    // 2. Create CRD resource via K8s API
    // 3. Create associated services and secrets
  }

  async scale(instanceId: string, config: ScaleConfig): Promise<void> {
    // 1. Patch CRD spec with new resource requirements
    // 2. Monitor operator reconciliation
  }

  async backup(instanceId: string, backupId: string, storagePath: string): Promise<void> {
    // 1. Create engine-specific backup job
    // 2. Upload to RustFS
  }

  async restore(instanceId: string, backupId: string, storagePath: string): Promise<void> {
    // 1. Create RestoreJob CRD
    // 2. Monitor restore completion
  }

  async deprovision(instanceId: string, namespace: string): Promise<void> {
    // 1. Delete CRD resource
    // 2. Clean up PVCs, secrets, services
    // 3. Delete namespace if empty
  }

  async rollingRestart(instanceId: string, namespace: string): Promise<void> {
    // 1. Annotate CRD with restart trigger
    // 2. Operator handles rolling restart
  }
}
```

### Step 4: Create Operator Manifest

Create an operator manifest at `/operators/{engine}-cluster.yaml` with the engine-specific configuration.

### Step 5: Register in Engine Registry

Edit `/services/dbaas-api/src/routes/engines.ts` to add the engine to the `AIDD_ENGINES` or `COMMERCIAL_ENGINES` array:

```typescript
{
  engine: DBEngine.NewEngine,
  displayName: 'Engine Display Name',
  description: 'Description of the engine.',
  versions: ['1.0', '1.1'],
  operator: 'operator-name',
  supportedHAModes: [HAMode.Standalone, HAMode.HA],
  profile: 'both',  // or PolicyProfileMode.CommercialFlexible
},
```

### Step 6: Update Policy Profiles

If the engine is AIDD-approved, add it to `ErpAiddStrictProfile.ALLOWED_ENGINES` in `/services/dbaas-api/src/policies/erp-aidd-strict.ts`.

If the engine is commercial-only, add it to `CommercialFlexibleProfile.ALLOWED_ENGINES` and also add it to `ErpAiddStrictProfile.DENIED_ENGINES` (with any aliases).

### Step 7: Write Tests

- Unit tests for the adapter (`operators/{engine}-adapter.test.ts`).
- Integration test for provisioning the engine.
- Policy validation tests confirming the engine is allowed/denied under each profile.

### Step 8: Update Documentation

- Add engine to the engine registry in API documentation.
- Update the changelog.
- Add engine-specific operational notes to runbooks.

## Plugin Development Guide

Plugins extend ERP-DBaaS with custom database engine support via the gRPC lifecycle interface. Plugins are available only under the `COMMERCIAL_FLEXIBLE` profile.

### Prerequisites

- A Kubernetes service account in the target cluster.
- A gRPC server implementing the `dbaas.plugin.v1.LifecycleService`.
- Network connectivity from the `dbaas-system` namespace to the plugin's gRPC endpoint.

### Step 1: Implement the gRPC Interface

Implement all 8 methods of the `LifecycleService`:

| Method | When Called | Required |
|---|---|---|
| `GetCapabilities` | On validation | Yes |
| `OnProvision` | After CRD creation | Yes |
| `OnScale` | After scale spec applied | Yes |
| `OnBackup` | When backup triggered | Yes |
| `OnRestore` | When restore triggered | Yes |
| `OnRotateCredentials` | When rotation requested | Yes |
| `OnDeprovision` | Before CRD deletion | Yes |
| `HealthCheck` | Periodic (every 30s) | Yes |

### Step 2: Deploy to Kubernetes

Deploy the plugin as a Kubernetes Deployment with a ClusterIP Service:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-engine-plugin
  namespace: dbaas-plugins
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-engine-plugin
  template:
    metadata:
      labels:
        app: my-engine-plugin
    spec:
      serviceAccountName: my-engine-plugin-sa
      containers:
        - name: plugin
          image: myregistry/my-engine-plugin:1.0.0
          ports:
            - containerPort: 50051
              protocol: TCP
          securityContext:
            runAsNonRoot: true
            readOnlyRootFilesystem: true
            allowPrivilegeEscalation: false
---
apiVersion: v1
kind: Service
metadata:
  name: my-engine-plugin
  namespace: dbaas-plugins
spec:
  selector:
    app: my-engine-plugin
  ports:
    - port: 50051
      targetPort: 50051
      protocol: TCP
```

### Step 3: Register the Plugin

```bash
curl -X POST http://localhost:8090/v1/dbaas/plugins \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{
    "name": "my-custom-engine",
    "engine": "myengine",
    "version": "1.0.0",
    "grpcEndpoint": "http://my-engine-plugin.dbaas-plugins.svc.cluster.local:50051",
    "capabilities": ["provision", "scale", "backup", "restore", "rotate", "health"],
    "securityContext": {
      "serviceAccount": "my-engine-plugin-sa",
      "namespace": "dbaas-plugins"
    }
  }'
```

### Plugin Naming Rules

- 3-64 characters.
- Must match regex: `^[a-z0-9][a-z0-9-]*[a-z0-9]$`.
- Must be globally unique.

### Plugin Security Requirements

Under `ERP_AIDD_STRICT` profile, plugins:
- Cannot extend denied engines (PostgreSQL, MySQL, MariaDB, Redis/Valkey).
- Cannot declare forbidden capabilities: `external_network_access`, `disable_security`, `raw_filesystem_access`, `privilege_escalation`.

Under `COMMERCIAL_FLEXIBLE` profile, plugins:
- Cannot declare forbidden capabilities: `privilege_escalation`, `host_network_access`, `host_pid_access`.

### Plugin Lifecycle States

```
pending_validation --> validated --> active
                  \              \
                   --> rejected    --> disabled
```

## Testing Requirements

All contributions must include appropriate tests. The testing pyramid for ERP-DBaaS:

### Unit Tests (Required for all PRs)

- **Location**: Co-located with source files (e.g., `provisioner.test.ts` next to `provisioner.ts`).
- **Framework**: Vitest.
- **Coverage requirement**: 80% line coverage for new code.
- **What to test**:
  - Zod schema validation (valid and invalid inputs).
  - Policy profile validation (allowed/denied engines, config validation).
  - Controller logic with mocked dependencies.
  - Database query construction.
  - Error handling paths.

```bash
# Run unit tests
make test

# Run with coverage
cd services/dbaas-api && npx vitest --coverage
```

### Integration Tests (Required for operator adapters)

- **Location**: `services/dbaas-api/tests/integration/`.
- **Requirements**: Running YugabyteDB and DragonflyDB instances.
- **What to test**:
  - Full request lifecycle (provision -> check status -> scale -> backup -> restore -> decommission).
  - Tenant quota enforcement.
  - Rate limiting behavior.
  - Credential rotation flow.

```bash
make test-integration
```

### E2E Tests (Required for release)

- **Location**: `tests/e2e/`.
- **Requirements**: Full Docker Compose environment or Kubernetes cluster.
- **What to test**:
  - Gateway -> API -> Database round-trip.
  - Authentication flow with JWT tokens.
  - CRD creation and status updates.
  - Backup/restore with actual RustFS.

```bash
make test-e2e
```

## Code Review Checklist

Reviewers should verify:

- [ ] TypeScript strict mode: No `any` types unless justified.
- [ ] Error handling: All route handlers have try/catch with `next(err)`.
- [ ] Validation: All user input validated with Zod schemas.
- [ ] Tenant isolation: All database queries filter by `tenant_id`.
- [ ] AIDD compliance: No references to denied engines in strict profile paths.
- [ ] SQL injection: All queries use parameterized statements (`$1`, `$2`).
- [ ] Tests: Unit tests included with adequate coverage.
- [ ] Logging: Structured log entries with relevant context (instanceId, tenantId, requestId).
- [ ] Documentation: JSDoc on exported functions/interfaces.
- [ ] Security headers: No removal or weakening of security headers.
- [ ] Rate limit headers: Included on relevant responses.
- [ ] Idempotency: Migrations use `IF NOT EXISTS`.

## Release Process

1. **Feature freeze**: All features merged to `develop` branch.
2. **Release branch**: Create `release/v{version}` from `develop`.
3. **QA testing**: Run full E2E test suite.
4. **Documentation update**: Update changelog, API docs, and version numbers.
5. **Tag and release**: Merge to `main`, tag `v{version}`, build Docker images.
6. **Deployment**: Rolling update via Helmfile.
7. **Verification**: Smoke tests in staging, then production.
